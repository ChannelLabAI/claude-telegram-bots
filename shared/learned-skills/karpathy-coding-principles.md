# Karpathy Coding Principles

> 蒸餾自 Andrej Karpathy 的 coding 行為準則。
> Source: https://github.com/forrestchang/andrej-karpathy-skills
> 適用對象：所有 Builder bot（Anna、三菜、Eric/KKKK）

**取捨說明：** 這些準則偏向謹慎而非速度。對於簡單任務，自行判斷是否適用。

---

## 1. 先思考，再動手（Think Before Coding）

**不要假設。不要藏住疑惑。把取捨攤出來。**

實作前：
- 明確說出你的假設。不確定就問。
- 如果有多種解讀方式，把它們列出來——不要自己默默選一個。
- 如果有更簡單的做法，說出來。該推回去就推回去。
- 如果有不清楚的地方，停下來。說出哪裡不清楚。問。

## 2. 簡單優先（Simplicity First）

**用最少的 code 解決問題。不加任何猜測性的東西。**

- 不加任何沒被要求的功能。
- 一次性用到的 code 不做抽象。
- 沒被要求的「彈性」或「可配置性」一律不加。
- 不可能發生的場景不做 error handling。
- 如果你寫了 200 行但 50 行就夠，重寫它。

自問：「資深工程師會說這太複雜嗎？」如果是，就簡化。

## 3. 手術刀式修改（Surgical Changes）

**只碰必須碰的。只清理自己製造的亂。**

修改現有 code 時：
- 不「順便改善」旁邊的 code、註釋或格式。
- 不重構沒有壞掉的東西。
- 保持現有風格，即使你有更好的寫法。
- 發現不相關的 dead code，提出來——但不要刪。

你的改動製造了孤兒時：
- 移除「你的改動」讓它沒用到的 import / 變數 / function。
- 不移除既有的 dead code（除非被要求）。

測試標準：每一行改動都要能追溯回使用者的要求。

## 4. 目標導向執行（Goal-Driven Execution）

**定義成功標準。驗證達到為止。**

把任務轉換成可驗證的目標：
- 「加驗證」→「為無效輸入寫測試，然後讓它通過」
- 「修 bug」→「寫能重現 bug 的測試，然後讓它通過」
- 「重構 X」→「確保重構前後測試都通過」

多步驟任務要先說計劃：
```
1. [步驟] → 驗證：[確認方式]
2. [步驟] → 驗證：[確認方式]
3. [步驟] → 驗證：[確認方式]
```

強的成功標準讓你能自己跑完迴圈。弱的標準（「讓它跑起來」）需要不斷確認。

---

**這些準則有效的跡象：** diff 裡不必要的改動變少、因過度複雜而重寫的情況減少、釐清問題的提問發生在實作之前而不是出錯之後。
